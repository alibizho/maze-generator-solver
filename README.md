# Maze Generator & Solver — Описание проекта

## Что это за проект

Генератор и решатель лабиринтов в терминале, написанный на C. Один человек
генерирует лабиринт и сохраняет его, другой решает лабиринт и отрисовывает
результат. Обе части соединяются через общий заголовочный файл (`maze.h`),
так что большую часть недели вы можете работать независимо, не блокируя
друг друга.

---

## Структура файлов

```
maze.h              общие структуры/типы + объявления функций
maze_generate.c     код Человека A
maze_solve.c         код Человека B (решение + отрисовка)
main.c               код Человека B (меню + связывание всего вместе)
Makefile             собирает всё вместе
```

`maze.h` — это общий контракт. В нём находятся определения структур и
*объявления* (не тела) всех функций, которые могут понадобиться вызвать из
другого файла. Оба `.c` файла подключают `#include "maze.h"` в начале.

```c
// maze.h
#ifndef MAZE_H
#define MAZE_H

#define MAX_SIZE 51

typedef struct {
    int rows, cols;
    int grid[MAX_SIZE][MAX_SIZE];
} Maze;

typedef struct {
    int row, col;
} Point;

typedef struct {
    Point path[MAX_SIZE * MAX_SIZE];
    int length;
    int cells_visited;
} SolveResult;

// функции Человека A (определены в maze_generate.c)
Maze create_maze(int rows, int cols);
void generate_maze(Maze *m, int seed);
int save_maze(Maze *m, const char *filename);
Maze load_maze(const char *filename);

// функции Человека B (определены в maze_solve.c)
SolveResult solve_bfs(Maze *m, Point start, Point end);
void print_maze(Maze *m);
void print_path_overlay(Maze *m, SolveResult *r);

#endif
```

**Почему это важно:** каждый день вы трогаете только свой собственный `.c`
файл. `maze.h` вы трогаете вместе, и только когда добавляете или меняете
общую структуру/функцию, в идеале это происходит один раз, в самом начале,
а не постоянно. Каждый из вас может собирать и тестировать свои функции до
того, как код другого человека вообще появится, поскольку вы пишете код,
ориентируясь на объявления в заголовочном файле, а не на ещё не готовую
реализацию другого человека.

---

## Как собрать проект

```
gcc -o maze main.c maze_generate.c maze_solve.c -Wall
```

Или через Makefile (рекомендуется, чтобы не перепечатывать это каждый раз):

```makefile
maze: main.c maze_generate.c maze_solve.c maze.h
	gcc -o maze main.c maze_generate.c maze_solve.c -Wall

clean:
	rm -f maze
```

Затем просто запускайте `make`.

## Человек A — Генерация + Данные + Сохранение

### Концепция

**`generate_maze` — это рекурсивный backtracking, то есть DFS с небольшим
отличием.** Вместо того чтобы просто отмечать узлы как посещённые, как в
обычном DFS по графу, вы *прокладываете проход* по мере движения:

1. Начинаете с клетки, отмечаете её как посещённую
2. Берёте её 4 потенциальных соседа (вверх/вниз/влево/вправо),
   **перемешиваете порядок**, чтобы лабиринт не выглядел предсказуемым
3. Для каждого непосещённого соседа: убираете стену между текущей клеткой
   и этим соседом (устанавливаете в 0 клетку грида с нечётным индексом
   между ними), затем **рекурсивно вызываете функцию из этого соседа**
4. Когда у клетки больше не остаётся непосещённых соседей, функция
   возвращается, это и есть backtrack. Рекурсия естественным образом
   разворачивается назад к предыдущей клетке, у которой остались
   неисследованные соседи.

`save_maze`/`load_maze` — это не алгоритмы, а просто сериализация: записать
грид в текстовый файл в согласованном формате, прочитать его обратно в ту
же структуру. Зафиксируйте точный формат файла с Человеком B до начала
написания кода (см. ниже), так как его код в итоге будет читать
сохранённые вами лабиринты.

`compute_stats` (низкий приоритет, делать в последнюю очередь) — это
простой цикл, считающий открытые стены на каждую комнату (её "степень").
Комната с только одной открытой стеной — это тупик. Не представляет
алгоритмического интереса, пропустите сначала, если не хватает времени.

### Функции, за которые вы отвечаете

```c
Maze create_maze(int rows, int cols);
// Выделить память/инициализировать грид. Всё начинается как стена (1).
// rows/cols относятся к КОМНАТАМ, а не к размеру грида — размер
// (2*rows+1) x (2*cols+1) вычисляется внутри функции.

void generate_maze(Maze *m, int seed);
// Рекурсивный backtracking. Используйте srand(seed), чтобы результаты
// были воспроизводимы при тестировании.

int save_maze(Maze *m, const char *filename);
// Записать лабиринт в файл. Вернуть 1 при успехе, 0 при ошибке.

Maze load_maze(const char *filename);
// Прочитать лабиринт из файла в том же формате, в котором его записал
// save_maze.

MazeStats compute_stats(Maze *m);   // низкий приоритет / доп. задача
// Подсчитать тупики, вычислить средний коэффициент разветвления.
```

### Предлагаемый формат файла (согласуйте с Человеком B до написания кода)

```
Строка 1: rows cols
Следующие (2*rows+1) строк: сам грид, один символ на клетку
  '#' = стена, ' ' или '.' = открыто
```

### На что обратить внимание

- **Ошибки на единицу в индексации (off-by-one)** — самая частая ошибка
  здесь. Комната в логической позиции `(r,c)` находится в гриде по адресу
  `(2r+1, 2c+1)`. Стена между комнатой `(r,c)` и комнатой справа от неё
  находится по адресу `(2r+1, 2c+2)`. Сразу напишите небольшую
  функцию-помощник или макрос для этого преобразования, а не вычисляйте
  это вручную везде по коду.
- Глубина рекурсии растёт пропорционально количеству комнат, для обычных
  размеров это не проблема, но не тестируйте очень большие лабиринты без
  учёта ограничений стека.
- Распечатывайте свой собственный вывод и проверяйте его глазами ещё до
  того, как появится `print_maze` от Человека B, так легче поймать
  поломанные стены на раннем этапе.

## Человек B — Решение + Отрисовка + Меню/Main

### Концепция

**`solve_bfs` — это поиск в ширину (Breadth-First Search).** Исследуйте
лабиринт расширяющимися "кольцами": сначала всё, что на 1 шаг от старта,
затем всё, что на 2 шага, и так далее, пока не дойдёте до финиша. Это
гарантирует самый короткий путь, так как вы никогда не посещаете более
дальнюю комнату раньше более близкой.

Вам нужны три вспомогательные структуры данных:
- **Очередь** комнат для обработки (реализуйте сами, обычного массива с
  индексами `front`/`back` достаточно)
- **Грид посещённых клеток**, параллельный лабиринту, чтобы не добавлять
  одну и ту же комнату в очередь два раза
- **Грид "пришёл из"**: для каждой комнаты запоминаете, из какой комнаты
  вы в неё попали при первом посещении

Алгоритм:
1. Добавить в очередь стартовую комнату, отметить как посещённую,
   `came_from[start] = none`
2. Пока очередь не пуста: взять из очереди комнату, проверить её 4 соседей
   (только тех, между которыми и текущей комнатой **нет стены**, здесь вы
   читаете данные грида от Человека A), для каждого непосещённого соседа:
   отметить как посещённого, установить `came_from[neighbor] = current`,
   добавить в очередь
3. Остановиться, когда финишная комната будет извлечена из очереди (или
   добавлена в неё)
4. **Восстановить путь**: начиная с финиша, многократно смотреть
   `came_from`, идти назад, пока не дойдёте до старта. Вы прошли путь в
   обратном порядке, поэтому развернуть список перед тем, как вернуть его.

Мысленная модель для шага "пришёл из": вы не строите путь во время поиска,
вы оставляете "хлебные крошки", а затем идёте по ним назад, когда уже
знаете, что дошли до цели.

**`solve_dfs`** (дополнительная задача) структурно похожа на функцию
генерации Человека A, та же рекурсивная идея, только вы проверяете, "есть
ли здесь уже открытый путь", и не пересекаете стены, останавливаясь, когда
доходите до финиша. Не гарантирует самый короткий путь, хорошо подходит
для сравнения BFS и DFS в вашей демонстрации.

**Функции отрисовки** — это обход грида и форматирование вывода, не
алгоритмы. Проходите по каждой клетке, печатаете символ в зависимости от
того, стена это, открытая клетка, старт, финиш или часть пути.
**Анимация** — то же самое, но повторяется: очистить терминал, перерисовать
с подсвеченными недавно исследованными клетками, сделать небольшую паузу,
повторить.

### Функции, за которые вы отвечаете

```c
SolveResult solve_bfs(Maze *m, Point start, Point end);
// Самый короткий путь через BFS + восстановление пути через came-from.

SolveResult solve_dfs(Maze *m, Point start, Point end);   // доп. задача
// Решение через DFS, для сравнения с BFS.

void print_maze(Maze *m);
// Статичная отрисовка лабиринта в терминал.

void print_maze_animated(Maze *m, SolveResult *r);
// Пошаговая визуализация поиска, с небольшой задержкой и очисткой экрана
// между кадрами (например, "\033[H\033[2J" + usleep()).

void print_path_overlay(Maze *m, SolveResult *r);
// Печать лабиринта с финальным решённым путём, нарисованным сверху.

int main(void);
// Цикл меню, связывающий все функции из обоих файлов. Именно здесь
// проявятся несоответствия между двумя половинами, поэтому собирайте и
// тестируйте это постепенно, а не всё разом в самом конце.
```

### На что обратить внимание

- Правильно реализуйте логику пустой/полной очереди, это самая частая
  ошибка в BFS.
- Не забудьте развернуть путь после того, как прошли его назад через
  `came_from`.
- Напишите `print_maze` на раннем этапе, ещё до того, как будет готов BFS.
  Человеку A это нужно, чтобы визуально отлаживать генерацию, а вам это
  понадобится, чтобы отлаживать свой решатель на вручную составленном
  тестовом лабиринте ещё до того, как появится генерация.
- Лабиринт, созданный вашим собственным генератором, всегда должен иметь
  путь между любыми двумя комнатами. Если `solve_bfs` когда-либо сообщает
  "путь не найден" на лабиринте, созданном генератором, это ошибка в
  генерации или в логике проверки стен, а не ожидаемый результат,
  разбирайтесь в причине, а не считайте это нормальным.

## Как работать над проектом, день за днём

1. **День 1, вместе:** согласуйте `maze.h` именно так, как показано выше,
   и согласуйте формат файла для save/load. Напечатайте его вместе, чтобы
   вы оба смотрели на одну и ту же версию. Не меняйте его в одиночку
   позже, сначала напишите напарнику, если что-то нужно изменить.
2. **Работайте independently в своём `.c` файле.** Вам не нужна реализация
   другого человека, чтобы начать, достаточно объявлений в заголовочном
   файле, чтобы начать писать и тестировать свои функции.
3. **Используйте git**, если возможно (`git add`, `git commit`, `git push`,
   `git pull`). Он автоматически справляется с ситуацией "мы редактировали
   разные файлы" и даёт вам историю изменений на случай проблем. Общая
   папка тоже сработает, если git не вариант, но потребует больше ручной
   координации.
4. **Интегрируйтесь рано, маленькими шагами, а не всё разом в конце:**
   - *Рано:* сырой лабиринт от Человека A корректно печатается через
     `print_maze` Человека B, это подтверждает, что соглашение по гриду
     одинаково с обеих сторон.
   - *В середине:* `solve_bfs` от Человека B находит правильный путь в
     лабиринте, сгенерированном Человеком A, это подтверждает, что логика
     проверки стен правильно читает грид Человека A.
   - *Позже:* весь конвейер работает через меню в `main()` без сбоев,
     генерация, решение, анимация, сохранение, загрузка — всё доступно из
     одного меню.
5. **Компилируйте часто.** Запускайте `make` (или команду `gcc` выше)
   каждый раз, когда заканчиваете функцию, а не только в конце дня, чтобы
   первый признак несоответствия структур или опечатки проявился сразу, а
   не после нескольких часов несвязанной работы поверх него.

Слияние двух непротестированных половин в последний день — самый частый
способ, которым проекты с лабиринтами ломаются прямо перед дедлайном,
контрольные точки интеграции выше существуют именно для того, чтобы это
предотвратить.
